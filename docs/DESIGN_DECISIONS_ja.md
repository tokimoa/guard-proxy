# Guard Proxy - 設計判断書

各設計判断について「背景」「選択肢」「判断」「理由」「トレードオフ」「変更可能性」を記録します。

## DD-01: プロキシ方式 — 独自FastAPI透過プロキシ

### 背景
npm/PyPI/RubyGemsのプロキシとして、既存OSS（Verdaccio, devpi等）を利用するか、独自実装するかを判断する必要がある。

### 選択肢
1. **Verdaccio（npm）+ devpi（Python）をfork/プラグイン拡張**
2. **独自のFastAPIベース透過プロキシ**
3. **MITM Proxy（mitmproxy等）でHTTPレベルでインターセプト**

### 判断
**選択肢2: 独自のFastAPIベース透過プロキシ**

### 理由
- VerdaccioはNode.jsエコシステムのツールであり、Pythonプロジェクトに組み込むと技術スタックが分裂する
- devpiはPyPI専用であり、npm/PyPI/RubyGemsの全てを統一アーキテクチャで扱えない
- FastAPIの`httpx`で上流レジストリへリクエストを中継する方が、スキャンパイプラインへの統合が自然
- mitmproxyは汎用すぎて、レジストリAPIのセマンティクスを理解した判定ができない

### トレードオフ
- npm registry API、PyPI Simple API、RubyGems Compact Index APIのプロトコル互換性を自前で保証する必要がある
- 特にnpmのtarball URLリダイレクト処理やスコープ付きパッケージ（@scope/pkg）の扱いに注意が必要
- 既存ツールの成熟したキャッシュ機構やUIを利用できない

### 変更可能性
低い。アーキテクチャの根幹に関わるため、途中での変更コストは高い。ただしnpm/PyPI/RubyGemsのAPIは安定しており、互換性の維持は現実的。

## DD-02: スキャナアーキテクチャ — Pipeline + Protocol

### 背景
複数のスキャン手法（cooldown, 静的解析, LLM）をどのように構成・組み合わせるかを決定する。

### 選択肢
1. **モノリシック: 1つの関数で全チェックを実行**
2. **Pipeline + Protocol: 各スキャナを独立モジュール化し順次実行**
3. **Plugin: 外部プラグインとして動的ロード**

### 判断
**選択肢2: Pipeline + Protocol**

### 理由
- 各スキャナの責務が明確に分離される
- 新しいスキャナの追加時に既存コードを変更する必要がない（Open/Closed Principle）
- パイプライン順序を設定で制御でき、特定のスキャナの無効化も容易
- Python Protocolクラスによる型安全性

### トレードオフ
- Pluginほどの柔軟性はない（外部開発者が独自スキャナを追加するにはコード変更が必要）
- パイプラインの順序が結果に影響しうる（前段のスキャナでfailした場合、後段をスキップするかどうか）

### 変更可能性
中程度。Protocol → Pluginへの拡張は比較的容易。

## DD-03: 判定エンジン — 重み付きスコアリング

### 背景
複数のスキャナ結果を統合して最終判定（allow/deny/quarantine）を下す方法を決定する。

### 選択肢
1. **ルールベース: いずれかのスキャナがfailなら即deny**
2. **重み付きスコアリング: 各スキャナの結果×重み×信頼度で計算**
3. **LLMに最終判定を委ねる**

### 判断
**選択肢2: 重み付きスコアリング**

### 理由
- ルールベース（選択肢1）では誤検知時にブロックが厳しすぎる
- LLM最終判定（選択肢3）ではコストと遅延が大きすぎる
- 重み付けにより、各スキャナの信頼度を反映した柔軟な判定が可能
- 閾値を設定ファイルで調整することで、ユーザーの環境に合わせたチューニングが可能

### スコア計算式

```
最終スコア = Σ (scanner_weight × verdict_score × confidence)

verdict_score:
  pass = 0.0
  warn = 0.5
  fail = 1.0

デフォルト重み（12スキャナ重み）:
  ioc:              1.0
  advisory:         0.9
  cooldown:         0.3
  metadata:         0.5
  maintainer:       0.4
  static_analysis:  0.4
  heuristics:       0.3
  ast:              0.4
  yara:             0.6
  reachability:     0.3
  license:          0.3
  dependency:       0.3
  llm_judge:        0.3

判定:
  allow:      スコア < 0.3
  quarantine: 0.3 ≤ スコア < 0.7
  deny:       スコア ≥ 0.7
```

### トレードオフ
- 重み・閾値のチューニングが必要（初期値の設定が難しい）
- 個別スキャナの「絶対deny」（例: 既知のマルウェア）はスコアリングをバイパスする必要がある

### 変更可能性
中程度。閾値の調整は容易だが、スコアリングモデル自体を変更する場合は判定エンジンの再設計が必要。

## DD-04: データベース — SQLite

### 背景
スキャン結果キャッシュ、IOCデータベース、監査ログの永続化先を決定する。

### 選択肢
1. **Redis**
2. **SQLite**
3. **PostgreSQL**
4. **ファイルベース（JSON/YAML）**

### 判断
**選択肢2: SQLite**

### 理由
- guard-proxyの主要ユースケースはローカル開発者（シングルユーザー）
- 外部依存なし（Pythonの標準ライブラリに同梱）
- WALモード有効化で読み書き並行処理に対応
- チーム共有の場合も数万パッケージ程度のキャッシュならSQLiteで十分
- バックアップはファイルコピーのみ

### トレードオフ
- 高頻度の同時書き込みには不向き（チームサーバーで大規模に使う場合）
- 全文検索やJSON型のサポートが限定的

### 変更可能性
高い。SQLAlchemy経由でアクセスするため、PostgreSQL等への移行は比較的容易。

## DD-05: LLMアーキテクチャ — マルチプロバイダー + 実行戦略パターン

### 背景
LLMスキャンのコスト・遅延と精度のバランスを、ユーザーの環境に応じて柔軟に調整可能にする必要がある。ローカルLLM（Ollama）の実用性が十分に高まった2026年の状況を踏まえ、クラウドAPIに依存しない運用も可能にしたい。

### 選択肢
1. **Claude API単一プロバイダー**
2. **クラウドLLMのみ複数プロバイダー（Claude + GPT）**
3. **ローカル + クラウドのマルチプロバイダー + 実行戦略**

### 判断
**選択肢3: マルチプロバイダー + 実行戦略パターン**

### 理由

#### OSS採用障壁の除去
- 「APIキーが必要」はOSSツールの最大の採用障壁の一つ
- `local_only`戦略により、APIキーなし・無料で全機能が使える
- M4 ProでQwen3.5-9Bが30-50 tok/sで動作し、スクリプトスキャン（数百行）は1-3秒で完了
- Ollama経由で6.6GBのメモリ消費、開発マシンで十分に実用的
- 代替モデル（Gemma4-26B, Qwen3-Coder-30B, Mistral Small 3.2）は32GB+のRAMユーザー向けにより高い精度を提供

#### コスト最適化
- `local_first`戦略により、大半のスキャンを$0で処理
- クラウドAPI呼び出しは全体の0.2〜1.5%に抑制可能
- 月額$0.5〜$2での運用が現実的

#### 精度の段階的保証
- ローカルLLM（9B）は明確なケース（安全/明らかに悪意あり）には十分な精度
- グレーゾーン（confidence < 0.8）のみクラウドにエスカレーションすることで、精度とコストを両立
- 難読化されたコードは静的解析のデオブフスケーターで前処理してからLLMに渡すことで、小規模モデルの弱点を補完

#### 統一インターフェース
- OllamaのOpenAI互換APIにより、プロバイダー間の統一が自然
- `LLMProvider` Protocolによる抽象化で、新プロバイダーの追加が容易
- vLLM等のカスタムサーバーも`OpenAIProvider`経由で対応可能

### 4つの実行戦略

| 戦略 | 動作 | ユースケース | コスト |
|---|---|---|---|
| `local_first` | ローカル → confidence低ならクラウドにエスカレーション | **推奨デフォルト** | $0.5〜$2/月 |
| `cloud_only` | クラウドLLMのみ使用 | Ollama未設定の環境 | $5〜$15/月 |
| `local_only` | ローカルLLMのみ使用 | オフライン/完全無料 | $0/月 |
| `consensus` | ローカル+クラウド両方 → 合意判定 | 高セキュリティ環境 | $5〜$15/月 |

### フォールバックチェーン

```
1. ローカルLLM (Ollama) が起動中 → 使用
2. Ollama未起動 or タイムアウト → クラウドプライマリ (Anthropic)
3. Anthropic API障害 → クラウドフォールバック (OpenAI)
4. 全LLM利用不可 → Static Analysisの結果のみで判定（degraded mode）
```

いずれの段階でも、LLM全滅時はstatic analysisの結果のみで判定を続行します。Guard Proxyはセキュリティツールであるため、LLM障害で全体が停止することは許容しません。

### デオブフスケーター連携

```
                  難読化スコア
                      │
    ┌─────────────────┼─────────────────┐
    │ 低〜中 (< 0.7)  │                 │ 高 (≥ 0.7)
    ▼                 │                 ▼
ローカルLLMで判定      │        クラウドLLMに直接送信
                      │        （小規模モデルでは精度不足の可能性）
```

難読化スコアが高いコードは、9Bモデルでは解析精度が落ちるため、ローカルをスキップしてクラウドに直接送信します。これはベンチマーク上の知見（小規模LLMは難読化コード解析に弱い）に基づく判断です。

### 構造化出力の保証

各プロバイダーごとに異なる構造化出力メカニズムを使用しますが、出力スキーマは統一します。

| プロバイダー | 構造化出力方式 | 信頼性 |
|---|---|---|
| Ollama | `format: json_schema` (GBNF文法ベース) | 7B以上で安定 |
| Anthropic | `tool_use` による構造化出力 | 高い |
| OpenAI | `response_format: json_schema` | 高い |

7B未満のモデル（Q3以下の量子化含む）ではJSON出力が不安定になる可能性があるため、`OLLAMA_MODEL`のデフォルトは`qwen3.5:9b`とし、ドキュメントで推奨最低サイズを明記します。

### トレードオフ
- プロバイダー間で判定結果にばらつきが出る可能性
  → consensus戦略で合意を取ることで軽減可能
  → 判定結果にprovider_nameを記録し、後から分析可能に
- ローカルLLMの精度がクラウドLLMに劣る場面がある
  → エスカレーション機構とデオブフスケーターで補完
  → `local_only`使用時はこのリスクを受容する前提
- 実装複雑度の増加
  → `LLMProvider` Protocolで統一し、各プロバイダーは独立したモジュール
  → 新プロバイダー追加は1ファイル追加のみ

### 変更可能性
高い。新プロバイダーの追加は`LLMProvider`を実装した1ファイルの追加のみ。戦略の追加も`judge.py`の拡張で対応可能。

## DD-06: デフォルト動作 — Warnモード（Enforceではない）

### 背景
ブロック時のデフォルト動作をどうするかを決定する。

### 選択肢
1. **Enforce（deny時にinstallをブロック）**
2. **Warn（deny時にログ警告のみ、installは許可）**

### 判断
**選択肢2: Warnモードをデフォルト**

### 理由
- 新規導入時に誤検知でinstallが止まると、開発者体験が悪化し導入を拒否される
- まずWarnモードで運用し、誤検知率を確認してからEnforceに切り替える運用が現実的
- 「入れてみたら開発が止まった」という体験は取り返しがつかない

### 設定
```
DECISION_MODE=warn     # 警告のみ（デフォルト）
DECISION_MODE=enforce  # ブロック有効
```

### 変更可能性
高い。環境変数1つで切り替え可能。

## DD-07: パッケージ解凍方針 — installスクリプト系ファイルのみ抽出

### 背景
パッケージのtarball/whlを全展開するか、必要なファイルのみ抽出するかを決定する。

### 選択肢
1. **全展開してスキャン**
2. **installスクリプト系ファイルのみ抽出**

### 判断
**選択肢2: installスクリプト系ファイルのみ抽出**

### 理由
- 大規模パッケージの全展開は数百MB〜数GBになり得る
- 攻撃の入り口はinstall時に自動実行されるファイルに限定される
- スキャン対象を絞ることで速度が大幅に改善される

### 抽出対象

#### npm
- `package.json` (scripts.postinstall, scripts.preinstall, scripts.install)
- 上記scriptsが参照するファイル

#### PyPI
- `setup.py`
- `setup.cfg`
- `pyproject.toml`
- `*.pth`
- 各パッケージの`__init__.py`

#### RubyGems
- `metadata.gz` → gemspecメタデータ（`extensions`配列の確認）
- `extconf.rb`（任意パス） — ネイティブ拡張ビルドスクリプト
- `ext/**/*.rb` — ネイティブ拡張ソース
- `rubygems_plugin.rb` — gem読み込み時に自動実行
- `Rakefile`

### トレードオフ
- install時以外のタイミング（import時等）で実行される悪意あるコードは検出できない可能性
  → PyPIの`__init__.py`スキャンで部分的に対応
- スクリプトが外部ファイルを参照する場合、参照先の解決が必要

### 変更可能性
中程度。全展開モードの追加は可能だが、パフォーマンスとのトレードオフ。

## DD-08: LLMデフォルト戦略 — `local_first`

### 背景
マルチプロバイダーLLM Judgeのデフォルト実行戦略を決定する。

### 選択肢
1. **`cloud_only` をデフォルト**（従来のSaaS型ツールと同様）
2. **`local_first` をデフォルト**（ローカル優先、必要時のみクラウド）
3. **`local_only` をデフォルト**（完全ローカル）

### 判断
**選択肢2: `local_first` をデフォルト**

### 理由
- OSSツールとして「インストールしたらすぐ使える」体験を重視
- Ollamaが入っていれば$0で動く、入っていなければクラウドにフォールバック
- `local_only`は精度面でのリスクが高い（エスカレーション不能）
- `cloud_only`はAPIキー必須で導入障壁が高い
- `local_first`は「Ollama推奨だがなくても動く」という最もフレキシブルな選択

### 自動検出
起動時にOllamaの可用性を自動検出し、以下のように動作します：

```
Ollama起動中 + モデルあり → local_first で動作
Ollama未起動 or モデルなし → cloud_only に自動フォールバック
クラウドAPI鍵も未設定      → static analysis only (LLM無効、警告表示)
```

### 変更可能性
高い。環境変数`LLM_STRATEGY`で即座に切り替え可能。

## DD-09: ライセンスコンプライアンススキャナ（v2.3.0）

### 背景
エンタープライズユーザーはサプライチェーンセキュリティポリシーの一環としてライセンスコンプライアンスチェックを必要とするが、ライセンスポリシーは組織ごとに大きく異なる。

### 選択肢
1. **組み込みの意見付き許可/拒否リスト**（例: AGPLは常にブロック）
2. **組み込みの意見を持たない、設定可能な許可/拒否リスト方式**
3. **ライセンススキャンなし（外部ツールに委ねる）**

### 判断
**選択肢2: 設定可能な許可/拒否リスト方式**

### 理由
- エンタープライズユーザーはそれぞれ異なるライセンスポリシーを必要とする。許容されるものは組織やプロジェクトによって異なる
- SPDXエクスプレッション解析とエイリアス正規化により、実環境に存在する多様なライセンス識別子に対応
- コピーレフト検出は個別の設定可能なアクション（warn vs. deny）として提供
- デフォルトはオプトイン: ユーザーがポリシーを設定するまでライセンススキャンは何もブロックしない

### トレードオフ
- ユーザーに初期設定の手間がかかる
- SPDXエクスプレッション解析の複雑さ（`MIT OR Apache-2.0` のような複合式）
- 一部のパッケージではライセンスメタデータが欠落または不正確

### 変更可能性
高い。ポリシーは完全に設定駆動。新しいライセンスカテゴリやアクションの追加は容易。

---

## DD-10: YARAルールマーケットプレイス（v2.4.0）

### 背景
YARAルールはパッケージコード内の既知の悪意あるパターンの検出に非常に効果的だが、脅威の状況は急速に変化する。コミュニティ提供のルールを取り込むメカニズムが必要。

### 選択肢
1. **バンドルルールのみ（Guard Proxyに同梱）**
2. **設定ファイルベースの外部ルールソース**
3. **URL + SHA256変更検出によるCLI管理マーケットプレイス**

### 判断
**選択肢3: CLI管理マーケットプレイス**

### 理由
- サプライチェーン攻撃検出にはルールの鮮度が重要。コミュニティルールがコアチームの維持範囲を超えてカバレッジを拡張する
- URL + SHA256変更検出によりルールの整合性を確保し、更新通知を可能にする
- CLI管理（`guard-proxy yara add/remove/update/list`）は設定ファイルの編集より使いやすい
- 起動時の自動更新はオプトイン機能として提供し、常に最新のルールを求めるユーザーに対応

### トレードオフ
- 信頼モデル: ユーザーが自らコミュニティルールソースを検証する必要がある
- ルール更新にネットワーク接続が必要（オフライン環境では手動インポートが必要）
- SHA256検証によりルール公開時にステップが追加される

### 変更可能性
中程度。マーケットプレイスプロトコル（URL + SHA256）はシンプルで拡張可能。新しいソースタイプ（Gitリポジトリ等）の追加は容易。

## DD-11: 到達可能性分析（v2.5.0）

### 背景
静的解析スキャナは、コード内に存在するが実際にはインストールフックやエントリポイントから到達不可能な危険な関数呼び出し（`eval`、`exec`、`child_process.exec`等）をフラグすることがある。これが偽陽性の原因となる。

### 選択肢
1. **到達可能性分析なし（すべての危険パターンをフラグ）**
2. **ファイル内ASTベースのコールグラフ分析**
3. **ファイル間のプログラム全体分析**

### 判断
**選択肢2: ファイル内ASTベースのコールグラフ分析**

### 理由
- 到達不可能な危険コードを「fail」ではなく「pass」とマークすることで偽陽性を削減
- PythonおよびJavaScriptのASTベース分析は高速でLLM不要
- ファイル内スコープは精度とパフォーマンスの実用的なトレードオフ
- ファイル間分析にはLLMティアのリソースが必要で、スキャン遅延が大幅に増加する

### トレードオフ
- ファイル間の呼び出しチェーンは追跡されない（別ファイルから間接的に呼ばれる危険な関数は到達不可能として検出されない）
- 動的ディスパッチやメタプログラミングは静的コールグラフ分析を無効化する可能性がある
- 言語サポートは当初PythonとJavaScriptに限定

### 変更可能性
中程度。新しい言語の追加にはその言語のAST解析の実装が必要。ファイル間分析への拡張は大きな作業であり、LLM統合が必要になる可能性が高い。

## DD-12: 検出強化

### 背景
Guard Proxyの成熟に伴い、攻撃者が良性のインジケータと悪意あるコードを組み合わせたり、正規表現のバックトラッキングを悪用するパターンを使用して検出を回避しようとする可能性がある。

### 選択肢
1. **すべてのインジケータを平等に扱う（良性インジケータがスコアを下げることを許容）**
2. **クリティカルパターンは決してダウングレードしない。高重大度の単一マッチ = fail**
3. **すべての曖昧なケースをLLMに委任**

### 判断
**選択肢2: クリティカルパターンは決してダウングレードしない**

### 理由
- セキュリティツールは検出側に傾くべき。偽陰性は偽陽性より悪い
- クリティカルパターン（例: 直接的な`eval(Buffer.from(...))`、C2通信）は、安全に見えるコードに囲まれていてもダウングレードすべきではない
- 高重大度の単一マッチでfail判定に十分
- 正規表現ウィンドウ縮小によるReDoS緩和は、プロキシ自体をパフォーマンス攻撃から保護する

### トレードオフ
- 攻撃コードに類似したパターンを正当に使用するパッケージの偽陽性率が増加する可能性
- 厳格なルールにより判定エンジンのコンテキスト考慮能力が低下
- 正規表現ウィンドウ縮小により、大きなコードブロックにまたがるパターンを見逃す可能性

### 変更可能性
低い。検出強化ルールの緩和は回避ベクターを生む危険がある。変更は徹底的なベンチマーク検証の上でのみ行うべき。
